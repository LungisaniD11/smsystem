This documentation summarizes the current implementation in the opened FutureTech Student Management System project, including architecture behavior in development and production environments.

1. Key Features

1.1 Secure Admin-Only Access

The system uses Google OAuth 2.0 authentication and then applies an admin allow-list policy. Only authenticated users who match configured admin emails or approved domains can access student management screens.

1.2 Complete Student Lifecycle Management

Administrators can create, update, search, soft delete, and permanently delete student records. The dashboard displays enrollment and operational metrics such as active/inactive students, recently added records, and image coverage.

1.3 Search and Pagination

Student lists support query-based search by names or identifiers with paginated views to maintain responsive performance and manageable navigation for larger datasets.

1.4 Profile Image Handling

Student profiles support image upload (JPEG and PNG). The platform validates file size, extension, MIME type, and binary magic bytes, then stores image content using local development storage or Azure Blob Storage in production scenarios.

1.5 Storage Flexibility for Different Environments

In development, the solution can automatically fall back to local JSON repository and local image storage when cloud settings are placeholders. In cloud-ready configuration, it uses Azure Cosmos DB for student documents and Azure Blob Storage for media assets.

1.6 Time-Limited SAS Access for Images

Instead of exposing permanent blob links, the application builds short-lived Shared Access Signature (SAS) URLs, reducing long-term link exposure and aligning with secure media retrieval practices.

1.7 Observability and Test Coverage

Application Insights telemetry is enabled for runtime monitoring, while xUnit tests are included for business services such as student management and admin access logic.

2. Security

2.1 Authentication and Authorization

  • Google OAuth 2.0 is configured as the challenge scheme.
  • Cookie authentication persists signed-in sessions.
  • An AdminOnly policy requires authenticated users with the Admin role.
  • StudentsController is protected with admin-only authorization attributes.

2.2 Admin Access Enforcement

Admin status is computed from normalized email and domain allow-lists. The service supports array and CSV configuration formats, making production settings easier to manage while preserving strict validation behavior.

2.3 Input and Upload Validation

  • Allowed image types are restricted to JPEG and PNG only.
  • Maximum upload size is configurable in BlobStorage options.
  • File signature checks (magic bytes) help block spoofed uploads.
  • Model validation and anti-forgery tokens protect data-entry actions.

2.4 HTTP and Browser Hardening

  • HTTPS redirection is enabled.
  • HSTS is enforced in non-development environments.
  • Security headers include: X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, Referrer-Policy, and Content-Security-Policy.

2.5 Secrets and Credential Practices

Sensitive values (Google secret, Cosmos key, Blob connection string) are intended to be managed via environment variables or Azure App Service settings rather than hard-coded in source files.

2.6 Data Access Security

Student profile images are served through temporary SAS links with configurable expiry windows. This limits unauthorized long-term access and supports least-privilege resource usage.

3. Deployment Information

3.1 Target Cloud Architecture

The deployment model is Azure-centric and includes App Service for hosting, Cosmos DB for student data, Blob Storage for images, and Application Insights for operational monitoring.

3.2 Required Azure Resources

  • Resource Group
  • App Service Plan and Web App (MVC application)
  • Cosmos DB account, database, and container
  • Storage account with student-images container
  • Application Insights linked to the web app

3.3 Configuration and Environment Variables

Deployment requires production values for authentication, admin access, Cosmos DB, and Blob Storage settings. The project supports nested application settings in Azure using double-underscore naming conventions.

3.4 Deployment Flow

  • Deploy first to a staging slot.
  • Verify login, student CRUD operations, uploads, and SAS image retrieval.
  • Swap staging with production for low-downtime releases.

3.5 Scaling and Monitoring

Autoscale policies can scale out under sustained high CPU and scale in when utilization drops. Application Insights should be configured with alerts for failed requests, exceptions, and latency thresholds.

3.6 Operational Checklist

  • Rotate credentials for Cosmos DB and Storage periodically.
  • Keep SAS expiry short for media links.
  • Use HTTPS-only endpoints and review security headers after deployment.
  • Validate admin allow-list entries whenever access teams change.
